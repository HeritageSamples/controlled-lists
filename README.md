# Heritage Samples Registry — SKOS Vocabularies

SKOS ConceptScheme representations of the controlled enum lists defined in the [HeritageSamples/schema](https://github.com/HeritageSamples/schema) JSON Schema repository.

## Background

The [Heritage Samples Registry](https://heritagesamples.org/) uses JSON Schema (Draft-07) to define its record types. Several properties within those schemas use `enum` arrays to restrict values to a fixed set — for example, `titleType`, `relationType`, and `descriptionType`. This repository exposes those fixed lists as standalone SKOS vocabularies so that they can be referenced, browsed, and linked independently of the schemas themselves.

## Source

All vocabulary terms are derived from version **v0.9** of the Heritage Samples Registry schemas. Only two schemas contain inline `enum` arrays:

| Source schema | Enum properties extracted |
|---|---|
| **HeritageObject** | `resourceType`, `titleType`, `descriptionType`, `objectType`, `funderIdentifierType`, `relatedIdentifierType`, `relationType`, `resourceTypeGeneral` |
| **Sample** | `titleType`, `descriptionType`, `sampleType`, `relatedIdentifierType`, `funderIdentifierType`, `relationType`, `resourceTypeGeneral`, `relatedIdentifiers/relationType`, `relatedIdentifiers/resourceTypeGeneral` |

The remaining schemas (`Person`, `Organisation`, `Material`, `VocabularyConcept`, `CordraUser`, `CordraGroup`, `Definitions`) do not define inline enum lists — they reference external controlled lists via `$ref`.

## File structure
```
enum-inventory.csv         Summary of all extracted enum lists
HeritageObject-*.jsonld    SKOS files derived from the HeritageObject schema
Sample-*.jsonld            SKOS files derived from the Sample schema
```

Each `.jsonld` file is a minimal SKOS `ConceptScheme` containing one `skos:Concept` per enum value. IRIs follow the pattern:
```
https://heritagesamples.org/vocab/{schemaName}/{propertyName}
https://heritagesamples.org/vocab/{schemaName}/{propertyName}/{slugifiedValue}
```

The `enum-inventory.csv` lists every generated file with its schema name, schema version, property name, number of values, and output filename.

## How these files were generated

The files were generated by running [`extract_enums_to_skos.py`](https://github.com/HeritageSamples/schema/blob/main/extract_enums_to_skos.py) from the root of the [HeritageSamples/schema](https://github.com/HeritageSamples/schema) repository:
```bash
git clone https://github.com/HeritageSamples/schema
cd schema
python3 extract_enums_to_skos.py
```

The script:
1. Locates the highest-version `v*.schema.json` file in each schema subfolder.
2. Recursively walks the full schema tree (including nested objects, array `items`, and `$defs`) looking for `enum` arrays.
3. Deduplicates entries that share the same property name and value set; where the same property name appears with different value sets (e.g. `relationType` in `Sample`), the second occurrence is given a path-qualified name.
4. Writes one JSON-LD file per enum list and a summary `enum-inventory.csv`.

No external Python dependencies are required beyond the standard library.

## Relationship to the source schemas

These vocabularies are derived artefacts. They will need to be regenerated whenever enum values are added, removed, or renamed in the source schemas. The `dc:source` field in each JSON-LD file records the schema name and version from which it was derived.
